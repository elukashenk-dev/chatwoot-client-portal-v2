# Chat Message Send UI Scenarios

Фактическая карта UI-поведения при отправке сообщений.

Основано на текущем коде portal frontend, а не на желаемом поведении:

- text send: `useOptimisticTextSend`, `useOfflineOutboxDrain`,
  `outboxDrain`, `chatClient`;
- attachment/voice send: `MessageComposer`, `useChatAttachmentSend`,
  `useVoiceRecorder`;
- transcript indicators: `MessageBubble`;
- header/notice: `ChatHeader`, `ChatRuntimeAlerts`.

Важно: в шапке при нормальной связи показывается не прямой network-status
телефона, а доступность поддержки: `На связи`, `Ответим позже`,
`Вне графика` или `Проверяем`. Если chat connection уже признан offline, это
перекрывается статусом `Нет связи`.

## Reliability, Customer Read And Typing Notes

Таблица ниже сохраняет фактическое текущее поведение текущего `main`.
Успешный backend-accepted статус называется `Отправлено`, а не `Доставлено`.

Customer-read и typing реализуются отдельным планом:
`docs/superpowers/plans/2026-06-04-customer-read-and-chat-typing.md`.

Этот план не добавляет две галочки пользователю портала для его исходящих
сообщений. Он синхронизирует обратную сторону: если пользователь портала реально
видит latest сообщения агента внизу ленты, backend сообщает Chatwoot через
`update_last_seen`, и агент может увидеть read status своих сообщений в
стандартной админке Chatwoot.

Ключевое правило будущей модели:

- `Отправлено` - backend/Chatwoot приняли сообщение;
- portal user-sent messages не получают fake `Прочитано поддержкой`;
- customer-read для агента - только private thread и только когда latest
  transcript видим внизу после render;
- group customer-read в Chatwoot dashboard не синхронизируется, потому что
  Chatwoot видит один group contact, а не каждого участника;
- push, app badge, unread counters и typing events не являются read source of
  truth.

| #   | Сценарий                                                           | Что видно в ленте                                                                                                               | Шапка и плашка                                                                                                                                         | Итоговое поведение                                                                                 |
| --- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------- |
| 1   | Связь есть, пользователь отправляет обычный текст                  | Сообщение сразу появляется в ленте со статусом `Отправляется`, после ответа backend меняется на `Отправлено` с галкой           | Если support online, в шапке `На связи`; плашки нет                                                                                                    | Сообщение подтверждается backend и остается как обычное отправленное                               |
| 2   | Связь есть, но backend отвечает медленно, меньше 20 секунд         | Сообщение висит как `Отправляется`                                                                                              | Шапка остается как при online; плашки `Нет связи` нет                                                                                                  | Если backend ответил успешно, статус становится `Отправлено`                                       |
| 3   | VPN/router cut: браузер еще думает, что online, text send завис    | Сначала `Отправляется`; через 20 секунд зависший text request abort-ится и сообщение становится `В очереди`                     | После abort: шапка `Нет связи`, плашка `Нет связи. 1 сообщение в очереди. Отправим, когда связь восстановится.`                                        | Сообщение остается в local outbox и потом досылается тем же `clientMessageKey`                     |
| 4   | Приложение уже знает, что связи нет, пользователь отправляет текст | Сообщение сразу появляется как `В очереди`                                                                                      | Шапка `Нет связи`, плашка с количеством сообщений в очереди                                                                                            | Текст сохраняется локально и отправится после восстановления связи                                 |
| 5   | Нет связи, но offline text queue недоступна или мало места         | Нового сообщения в ленте нет                                                                                                    | Под composer показывается ошибка про нехватку места/невозможность сохранить сообщение                                                                  | Draft остается в поле, отправка в outbox не происходит                                             |
| 6   | Сообщение было в очереди, связь восстановилась                     | `В очереди` может кратко перейти в `Отправляется`, затем исчезает optimistic bubble и появляется backend-сообщение `Отправлено` | Может появиться плашка `Связь восстановилась. Обновляем чат...`; затем плашка уходит                                                                   | Outbox record удаляется после успешной отправки                                                    |
| 7   | Backend вернул временную ошибку или rate limit при text send       | Сообщение остается `В очереди`                                                                                                  | Если это не network error, шапка не обязательно становится `Нет связи`                                                                                 | Outbox попробует отправить позже по retry/backoff или `Retry-After`                                |
| 8   | Backend отклонил text send по доступу/conflict                     | Сообщение получает статус `Не отправлено` с иконкой retry                                                                       | Если online, под сообщением кнопка `Повторить`; если offline, кнопка показывает `Нет сети`                                                             | Повтор доступен только для failed text messages без вложений                                       |
| 9   | Пользователь отправляет файл при online                            | Сначала виден preview файла в composer; отдельной optimistic bubble в ленте нет                                                 | Кнопка отправки становится состоянием `Отправляем`; плашки нет                                                                                         | После ответа backend файл появляется в ленте как отправленное сообщение с галкой                   |
| 10  | Пользователь пытается отправить файл, когда chat offline           | Кнопка прикрепления файла disabled; файл нельзя выбрать обычным путем                                                           | Шапка `Нет связи`, offline-плашка зависит от очереди текстов                                                                                           | Offline-очереди для файлов нет                                                                     |
| 11  | Файл уже выбран, а связь пропала до отправки                       | Preview файла остается в composer                                                                                               | Attachment send disabled; файл можно убрать из preview                                                                                                 | Файл не уходит в outbox, отправка файла ждет online-состояния                                      |
| 12  | File/attachment send завис из-за плохой сети                       | В ленте нового сообщения нет, composer остается в состоянии отправки                                                            | Специального 20-секундного timeout для attachment send сейчас нет                                                                                      | UI ждет завершения browser/backend request; при network error покажет offline-состояние или ошибку |
| 13  | Пользователь записывает голосовое при online                       | Во время записи показывается панель `Запись 00:xx`; после отправки голос идет как attachment                                    | Пока запись активна, обычные controls заблокированы                                                                                                    | После успеха в ленте появляется audio attachment с галкой                                          |
| 14  | Пользователь хочет записать голосовое при offline                  | Кнопка микрофона disabled                                                                                                       | Шапка `Нет связи`                                                                                                                                      | Offline-очереди для голосовых нет                                                                  |
| 15  | Микрофон недоступен или permission denied                          | Сообщение в ленте не создается                                                                                                  | Под composer показывается ошибка: например `Разрешите доступ к микрофону...`, `Микрофон не найден.` или `Голосовая запись недоступна в этом браузере.` | Пользователь должен исправить доступ/устройство и попробовать снова                                |
| 16  | Chat snapshot открыт из cache без связи                            | Сохраненные сообщения видны, новые тексты можно ставить в очередь, если offline storage доступен                                | Плашка `Нет связи. Показываем сохраненные сообщения.` или плашка с количеством очереди                                                                 | Read-only история доступна из cache, text outbox работает локально                                 |
| 17  | Чат недоступен или не готов к отправке                             | Composer disabled, placeholder `Чат временно недоступен`                                                                        | Может быть not-ready экран или offline/error notice                                                                                                    | Отправка сообщений не запускается                                                                  |

## Planned Customer Read And Typing Scenarios

Эти строки не описывают текущий код. Это целевая карта для будущего
customer-read and typing slice.

| #   | Сценарий                                                  | Что видно пользователю портала                                    | Что видит/получает агент в Chatwoot                                      | Итоговое правило                                                            |
| --- | --------------------------------------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------ | --------------------------------------------------------------------------- |
| R1  | Пользователь открыл private chat и находится внизу latest | Входящие agent messages видны; отдельного `Прочитано вами` нет    | Chatwoot получает contact `update_last_seen`; agent messages become read | Read sync идет только после visible-bottom latest transcript                |
| R2  | Пользователь открыл чат из offline cache                  | Сохраненные сообщения видны без read sync                         | Chatwoot не получает `update_last_seen`                                  | Cached/offline snapshot не является чтением для Chatwoot                    |
| R3  | Пользователь читает старую историю выше низа              | Новые agent messages не автоскроллят насильно                     | Chatwoot не получает `update_last_seen` до возврата вниз                 | Чтение истории не помечает новые сообщения агента прочитанными              |
| R4  | Пользователь внизу, приходит новое сообщение агента       | Лента auto-follow показывает новое сообщение                      | После render portal вызывает `update_last_seen`                          | Если latest реально видим, agent-side read обновляется быстро               |
| R5  | Пользователь отправил сообщение агенту                    | Исходящее сообщение остается `Отправлено` с одной галкой          | Агент получает обычное incoming message                                  | Нет portal-visible двух галочек без настоящего agent-read event             |
| R6  | Пользователь typing в private chat                        | В портале нет отдельной плашки про собственный typing             | Агент видит customer typing indicator                                    | Backend вызывает Chatwoot public `toggle_typing` через tenant authority     |
| R7  | Агент typing в Chatwoot                                   | Появляется `Сотрудник печатает...`, затем исчезает по off/timeout | Агент работает в стандартной админке                                     | Typing event transient; не создает message, push, unread или read receipt   |
| R8  | В group thread один участник открыл чат                   | В портале нет group read marker                                   | Chatwoot group contact read не синхронизируется                          | Нельзя честно показать per-user group read в стандартном Chatwoot dashboard |
| R9  | В group thread пользователь typing                        | В портале нет отдельной плашки про собственный typing             | Агент может видеть generic group/contact typing                          | Group typing не обещает индивидуального участника                           |
