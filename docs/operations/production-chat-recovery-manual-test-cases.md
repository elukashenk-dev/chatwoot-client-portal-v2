# Ручные тесткейсы восстановления чата на production

Этот чеклист проверяет production-поведение последнего блока восстановления
чата: переход в offline по фактической сетевой ошибке, проверку восстановления
соединения без надежного события `online`, таймаут зависших запросов
восстановления/отправки, отправку очереди из открытого приложения или через
Background Sync и предупреждения о состоянии связи.

Использовать после деплоя изменений в runtime чата, service worker,
offline-кеш, устойчивую очередь текстовых сообщений или восстановление PWA.

## Безопасность

- Проверять только на тестовом пользователе/контакте портала.
- Не записывать пароли, session cookies, Chatwoot-токены, текст приватных
  сообщений реальных клиентов или скриншоты с чувствительными данными.
- Все smoke-сообщения начинать с уникального префикса:
  `PROD-RECOVERY-YYYYMMDD-HHMM-<case-id>`.
- Перед кейсами с отправкой сообщений убедиться, что доступен agent/admin view
  Chatwoot для проверки доставки ровно одного сообщения.
- Если тест выполняется на реальном телефоне, фиксировать устройство, браузер и
  ОС, но не фиксировать персональные данные владельца устройства.

## Данные прогона

```md
| Поле                  | Значение |
| --------------------- | -------- |
| Дата/время            |          |
| Tenant URL            |          |
| Commit деплоя         |          |
| DEPLOY_SOURCE         |          |
| Пользователь портала  |          |
| Chatwoot account      |          |
| Chatwoot inbox        |          |
| Устройство/браузер    |          |
| Метод отключения сети |          |
```

## Предварительная проверка

1. Открыть `https://lk.provgroup.ru/api/health`.
2. Открыть `https://lk.provgroup.ru/api/tenant`.
3. На VM прочитать `/opt/chatwoot-client-portal-v2/DEPLOY_SOURCE.txt`.
4. Проверить `source_dirty=false` и ожидаемый commit.
5. Открыть приложение online, войти тестовым пользователем и открыть
   `private:me`.
6. Если приложение показывает `Доступна новая версия приложения`, нажать
   `Обновить` и заново открыть чат online до начала offline-кейсов.

Ожидаемо:

- health возвращает `status: ok`;
- tenant возвращает ожидаемый tenant slug;
- открытый чат показывает активный композер;
- нет пустого экрана или неконтролируемого JavaScript crash в консоли.

## PCR-001. Системный offline-сигнал показывает одно предупреждение

Тип: только чтение/offline
Риск: высокий

Шаги:

1. Открыть `private:me` online и дождаться готовности ленты и композера.
2. Перевести браузер/устройство в реальный offline-режим: авиарежим,
   offline-симуляция Chrome или отключение Wi-Fi/мобильного интернета.
3. Пока не отправлять сообщение.
4. Проверить шапку чата и предупреждение в интерфейсе.
5. Если платформа позволяет, перезагрузить страницу, оставаясь offline.

Ожидаемо:

- Шапка/присутствие показывает `Нет связи`.
- Предупреждение в интерфейсе показывает одно offline-сообщение для
  сохраненного чата: `Нет связи. Показываем сохраненные сообщения.`
- Ранее открытая лента остается видимой.
- Текстовый композер остается доступным для сообщения в очередь.
- Действия с файлами и голосом не доступны как устойчивые offline-отправки.
- Перезагрузка или повторное offline-открытие не зависает на native/web splash.

Считать провалом:

- Offline-предупреждение не появляется после подтвержденного offline-состояния.
- Одновременно видны несколько противоречивых предупреждений.
- Лента исчезает, хотя кеш был прогрет.
- Приложение зависает на PWA splash/startup screen.

## PCR-002. Отключение интернета на роутере ставит текст в очередь

Тип: отправка сообщения/offline
Риск: высокий

Это телефонный сценарий, где Wi-Fi остается подключенным, но роутер теряет
доступ в интернет. Приложение не должно зависеть от того, станет ли
`navigator.onLine` равным `false`.

Шаги:

1. Открыть `private:me` online и дождаться активного композера.
2. Оставить Wi-Fi подключенным.
3. Отключить upstream internet на роутере, заблокировать WAN или иначе сделать
   backend-запросы недоступными, пока устройство все еще считает Wi-Fi
   подключенным.
4. Отправить текст: `PROD-RECOVERY-<timestamp>-PCR-002`.
5. Подождать до 25 секунд.
6. Проверить ленту, шапку чата, предупреждение в интерфейсе и состояние кнопки
   отправки.

Ожидаемо:

- Пока запрос ожидает ответа, сообщение может показывать состояние отправки.
- После сетевой ошибки или таймаута отправки чат становится offline, даже если
  браузер все еще считает себя online.
- Шапка показывает `Нет связи`.
- Предупреждение в интерфейсе показывает:
  `Нет связи. 1 сообщение в очереди. Отправим, когда связь восстановится.`
- Сообщение остается видимым как queued, не теряется и не помечается sent
  раньше backend-подтверждения.
- Общая постоянная ошибка отправки не заменяет состояние очереди.

Считать провалом:

- Проходит больше 30 секунд без offline-индикации и без состояния очереди.
- Сообщение теряется из ленты.
- Сообщение помечается sent до подтверждения backend/Chatwoot.
- UI зависает или композер не восстанавливается.

## PCR-003. Проверка восстановления соединения работает без нового online-события

Тип: отправка сообщения/восстановление
Риск: высокий

Шаги:

1. Начать из состояния очереди, созданного в `PCR-002`.
2. Вернуть upstream internet на роутере, не переключая Wi-Fi телефона,
   авиарежим или offline/online-контролы браузера.
3. Оставить чат открытым и видимым.
4. Подождать до 45 секунд для проверки соединения в открытом приложении и
   отправки очереди.
5. Проверить сообщение в портале и в agent view Chatwoot.

Ожидаемо:

- Предупреждение об очереди исчезает после доказанной доступности backend.
- Текст из очереди отправляется автоматически без ручной перезагрузки.
- Финальная лента портала содержит ровно одно отправленное/canonical сообщение.
- Agent view Chatwoot содержит ровно одно совпадающее сообщение.
- Композер снова доступен.
- После успешного восстановления не остается устаревшего `Нет связи`.

Считать провалом:

- Сообщение остается queued до ручной перезагрузки.
- Сообщение дублируется в портале или Chatwoot.
- Offline-предупреждение остается после успешных backend-запросов.
- Восстановление зависит от ручного переключения сети устройства, чтобы
  принудительно получить событие `online`.

## PCR-004. Короткий обрыв во время отправки не дублирует сообщение

Тип: отправка сообщения/восстановление
Риск: высокий

Шаги:

1. Открыть чат online.
2. Начать отправку: `PROD-RECOVERY-<timestamp>-PCR-004`.
3. Сразу создать короткий обрыв сети, затем вернуть сеть через 5-10 секунд.
4. Дождаться стабильного состояния интерфейса.
5. Проверить ленту портала и agent view Chatwoot.

Ожидаемо:

- Сообщение либо отправляется нормально, либо кратко становится queued и затем
  уходит из очереди.
- Финальное сообщение ровно одно.
- UI не показывает одновременно queued и sent копии одного текста.
- Для краткого обрыва не показывается постоянное состояние ошибки.

Считать провалом:

- Появляются дубли сообщений.
- Отправленное сообщение остается продублированным локальным пузырем из очереди.
- Состояние retry/spinner зависает после восстановления сети.

## PCR-005. Сообщение в очереди живет после закрытия и offline-повторного открытия PWA

Тип: отправка сообщения/installed PWA
Риск: высокий

Шаги:

1. Установить/открыть PWA в Android Chrome или iOS/iPadOS Home Screen.
2. Войти online и открыть `private:me`.
3. Перейти полностью offline.
4. Отправить текст: `PROD-RECOVERY-<timestamp>-PCR-005`.
5. Проверить, что он видим как queued.
6. Полностью закрыть PWA из переключателя приложений.
7. Заново открыть PWA, оставаясь offline.
8. Вернуть сеть, пока PWA открыта.

Ожидаемо:

- Offline-повторное открытие показывает сохраненный чат, а не бесконечный
  splash.
- Текст в очереди остается после закрытия и повторного открытия приложения.
- После возвращения сети текст из очереди отправляется один раз.
- Финальная лента и Chatwoot содержат ровно одно совпадающее сообщение.

Считать провалом:

- Текст в очереди исчезает после полного закрытия приложения.
- Приложение открывается в blank/splash и не доходит до сохраненного чата.
- При включенном VPN приложение зависает на PWA splash/startup screen
  (`F-PWA-004`).
- Текст отправляется дважды после восстановления.

## PCR-006. Background Sync или отправка очереди при следующем открытии

Тип: отправка сообщения/installed PWA
Риск: высокий

Шаги:

1. В Android Chrome installed PWA открыть чат online.
2. Перейти offline.
3. Поставить текст в очередь: `PROD-RECOVERY-<timestamp>-PCR-006`.
4. Полностью закрыть PWA.
5. Вернуть сеть, пока PWA остается закрытой.
6. Подождать 1-3 минуты.
7. Проверить agent view Chatwoot.
8. Заново открыть PWA online.

Ожидаемо:

- Если Background Sync на платформе запускается, Chatwoot получает ровно одно
  сообщение, пока PWA закрыта.
- Если Background Sync на платформе не запускается, сообщение остается queued и
  отправляется при следующем открытии PWA, online-состоянии или visibility
  event.
- Оба варианта должны давать ровно одно финальное сообщение без локальной
  потери.

Примечание по платформам:

- iOS/iPadOS Home Screen PWA не имеет надежного Background Sync; background
  часть отмечать `BLOCKED` или `N/A`, затем проверять отправку очереди при
  следующем открытии.

Считать провалом:

- Сообщение теряется, пока приложение закрыто.
- Сообщение дублируется.
- Сообщение остается queued после повторного online-открытия.

## PCR-007. Приоритет предупреждений в интерфейсе

Тип: визуальная/runtime-проверка
Риск: средний

Шаги:

1. Открыть чат online.
2. Зафиксировать нормальное состояние шапки.
3. Перейти offline без текста в очереди.
4. Зафиксировать шапку и предупреждение.
5. Поставить одно текстовое сообщение в очередь.
6. Еще раз зафиксировать шапку и предупреждение.
7. Вернуть сеть и наблюдать переходное состояние.

Ожидаемо:

- Online-шапка показывает доступность поддержки или состояние проверки, а не
  фейковый сетевой статус.
- Offline без текста в очереди показывает предупреждение о сохраненном чате.
- Offline с текстом в очереди заменяет предупреждение о сохраненном чате на
  предупреждение с количеством сообщений в очереди.
- Во время восстановления может кратко появиться
  `Связь восстановилась. Обновляем чат...`.
- Одновременно видно только одно предупреждение в интерфейсе.

Считать провалом:

- Предупреждение о сохраненном чате и предупреждение об очереди показаны
  одновременно.
- Шапка говорит online/support-available, пока runtime считает чат offline.
- Восстановленное состояние продолжает показывать offline-предупреждение после
  успешных backend-запросов.

## PCR-008. Сохраненная старая история читается offline

Тип: только чтение/offline
Риск: средний

Шаги:

1. Открыть чат online.
2. Загрузить старые сообщения минимум до одной старой страницы.
3. Перейти offline.
4. Если платформа позволяет, перезагрузить или заново открыть чат.
5. Прокрутить к заранее прогретой старой истории.

Ожидаемо:

- Ранее прогретые старые сообщения доступны offline.
- Приложение не обещает бесконечную непрогретую историю во время offline.
- Возврат online не дублирует старые сообщения.

Считать провалом:

- Прогретые старые сообщения исчезают после offline-перезагрузки.
- Приложение блокирует весь чат из-за offline-запроса старой истории.
- Старые сообщения дублируются после восстановления соединения.

## PCR-009. Файлы и голос остаются только online

Тип: защита от offline-отправки медиа
Риск: средний

Шаги:

1. Открыть чат online.
2. Перейти offline.
3. Попробовать прикрепить файл.
4. Попробовать запись/отправку голоса, если устройство позволяет доступ к
   микрофону.
5. Для сравнения поставить обычное текстовое сообщение в очередь.

Ожидаемо:

- Текст можно поставить в очередь.
- Файл и голос не добавляются скрыто в устойчивую offline-очередь.
- UI держит media controls отключенными или явно заблокированными во время
  offline.
- После перезагрузки не остается зависшего частичного upload/audio-сообщения.

Считать провалом:

- Файл или голос выглядит queued как текст.
- Композер падает после выбора медиа offline.
- Частичный upload/audio state остается зависшим.


`BLOCKED` использовать только с конкретной причиной: нет тестового
пользователя, нет физического устройства, нет доступа к Chatwoot/agent view,
нет способа сделать router/WAN cut или платформа не поддерживает Background
Sync.

## Если кейс упал

Создать отдельный finding на каждую наблюдаемую проблему в `docs/findings/`.

Finding-файл должен содержать:

- id кейса;
- tenant URL;
- commit деплоя из `DEPLOY_SOURCE.txt`;
- устройство/браузер/ОС;
- точный метод отключения сети;
- шаги воспроизведения;
- ожидаемое и фактическое поведение;
- скриншоты/log notes без secrets;
- было ли сообщение доставлено, потеряно, задублировано или зависло.

Не чинить production smoke failure внутри несвязанного feature/docs scope.
