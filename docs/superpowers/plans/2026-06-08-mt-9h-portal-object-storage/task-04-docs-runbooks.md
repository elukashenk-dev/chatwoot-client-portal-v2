Зафиксировать product/ops decision в stable docs: storage является внутренней
частью portal production stack, а не задачей B2B-клиента.

- [ ] **Step 2: Update `D-014A` decision**

In `docs/architecture/decisions.md`, extend `D-014A` with:

```markdown
- production default: the portal deployment owns an internal S3-compatible
  MinIO service and a persistent Docker volume for branding binaries. This is
  part of the provider-operated portal stack, not a B2B-client provisioning
  responsibility. External managed S3-compatible storage is a future operator
  mode that requires a separate compose/profile/runbook slice; it is not part
  of the current default one-VM install path.
```

- [ ] **Step 4: Update clean reinstall runbook**

In `docs/operations/production-clean-reinstall.md`:

0. Add a new section before the reset step:

```markdown
## Data Safety For Portal Object Storage

After branding uploads are enabled, `portal-object-storage-data` is production
data alongside `portal-db-data`.

Default reinstall/deploy flow must preserve both volumes. Destructive reset is
allowed only when the operator explicitly confirms that portal-owned data is
disposable or has been backed up and restored elsewhere.
```

1. Rewrite the current reset step so the default path preserves production data:

```markdown
Default data-preserving reinstall:

- stop portal containers with `docker compose ... down --remove-orphans`;
- keep `chatwoot-client-portal-v2_portal-db-data`;
- keep `chatwoot-client-portal-v2_portal-object-storage-data`;
- remove/recreate app code, images and containers only.
```

Move the existing broad volume deletion command into a separate subsection:

```markdown
Destructive portal data reset:

Run this only when the operator explicitly confirms that portal-owned data is
disposable or after a verified backup/restore plan.

Before removing volumes, capture:

docker volume ls | grep '^chatwoot-client-portal-v2\_'

Then require a separate confirmation line in the runbook before running:

docker volume ls --format '{{.Name}}' \
 | grep -E '^chatwoot-client-portal-v2\_' \
 | xargs -r docker volume rm
```

Do not leave the broad `docker volume rm` command in the default reset path.

2. Add required generated storage inputs after Postgres secrets:

```text
PORTAL_OBJECT_STORAGE_ROOT_PASSWORD=<generated by installer>
BRANDING_ASSET_STORAGE_SECRET_ACCESS_KEY=<generated by installer>
```

3. In installer flow, add:

```markdown
4a. starts portal-owned object storage and initializes the branding asset bucket
and app user;
```

4. In Expected Env Shape, add:

```text
BRANDING_ASSET_STORAGE_ENDPOINT=http://portal-object-storage:9000
BRANDING_ASSET_STORAGE_BUCKET=portal-branding-assets
BRANDING_ASSET_STORAGE_ACCESS_KEY_ID=portal_v2_branding_assets
BRANDING_ASSET_STORAGE_FORCE_PATH_STYLE=true
```

5. In Verify Containers, add:

```markdown
- `portal-object-storage` running;
- `portal-object-storage-init` exited successfully;
- no backend errors about branding asset storage configuration.
```

6. In Acceptance Criteria, add:

```markdown
- portal-owned object-storage volume exists and is treated as production data;
- data-preserving reinstall path does not remove `portal-db-data` or
  `portal-object-storage-data`;
- destructive clean reset has an explicit warning and requires operator
  approval after backup;
- branding asset upload/readback works through portal admin and public asset
  routes.
```
