# MT-9H: Portal-Owned Object Storage For Branding Assets

## Goal

Make branding asset uploads production-ready for the intended B2B deployment
model: one VM runs Chatwoot plus the portal, and a new B2B client never has to
know about S3, buckets, object keys or storage credentials.

## Architecture

Branding asset storage remains S3-compatible and backend-owned, but the default
production deployment must include an internal portal-owned MinIO service. The
portal backend uses bucket-scoped app credentials generated by the installer,
not browser credentials and not Chatwoot storage. Portal Postgres remains the
source of truth for tenant ownership, active asset references and public asset
versions.

## Tech Stack

Docker Compose, MinIO, MinIO client (`mc`), Fastify backend env validation,
Node.js installer scripts, production runbooks, Vitest and code-health checks.

The gap found during local testing:

- local development supports MinIO through `pnpm storage:up`;
- production `.env.production.example` mentions S3-compatible storage;
- production compose and installer do not package object storage as an
  internal portal-owned service;
- production backend can start without storage env and fail only when an admin
  uploads an image.

This plan closes that deploy-readiness gap without changing the browser or
branding API contracts.

## Product Decision

Default production mode:

- one portal-owned MinIO service per portal deploy;
- one internal bucket for all tenants in that portal deploy;
- object keys remain tenant-prefixed;
- active asset ownership remains in portal Postgres with `tenant_id`;
- browser reads only through portal routes;
- MinIO ports are not published publicly in production compose;
- adding a new tenant requires no object-storage provisioning step.

External S3-compatible storage is deliberately deferred. This slice supports
one default production path only: internal portal-owned MinIO in the production
compose stack. Adding managed S3/Yandex/Selectel as an operator override would
need a separate compose/profile/runbook slice so it does not weaken the default
one-VM install.

## Acceptance Criteria

- Production compose includes internal `portal-object-storage` and
  `portal-object-storage-init` services.
- Production compose does not publish MinIO API or console ports to the host.
- Production backend receives complete `BRANDING_ASSET_STORAGE_*` env values.
- Production backend depends on object-storage bucket/user initialization.
- Installer generates root credentials, app credentials and bucket name.
- Backend uses app credentials for branding storage, not MinIO root
  credentials.
- Object-storage init resets the bucket policy and app user to match the
  current generated env, so `--reconfigure` cannot leave backend credentials
  out of sync with MinIO.
- In `NODE_ENV=production`, missing branding storage env fails during backend
  startup instead of returning upload-time `503`.
- Code-health fails if future compose/env edits remove the production storage
  wiring.
- Production docs describe storage as portal-owned internal infrastructure,
  not as a client responsibility.
- Production runbook includes upload/readback acceptance for branding assets.
- Existing local `pnpm storage:up` development path remains unchanged.
- Browser still never receives MinIO endpoint, bucket, object key or storage
  credentials.
- No Chatwoot core, Chatwoot DB or Chatwoot uploads/storage are changed.
- `MT-9H final branding QA/docs/deploy readiness` remains open until the final
  full branding QA/deploy closure is done.

## Non-Goals

- Do not add per-tenant buckets.
- Do not store branding files in backend/frontend container filesystem.
- Do not expose MinIO console as part of the product UI.
- Do not migrate Chatwoot uploads.
- Do not add image resizing or transformation.
- Do not change admin branding upload UI.
- Do not support external S3-compatible production storage in this slice.
- Do not mark all of `MT-9H` complete before final browser/PWA/cache QA and
  deploy docs are closed.
